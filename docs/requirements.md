## 01. 背景・目的

CrowdWorksでは「AI研修・AI講師」関連の募集が約80件確認されており、教材を毎回ゼロから作成するコスト（企画・台本執筆・スライド制作）を下げたいという需要が継続的に発生している。3件の参考案件はいずれも、テーマ入力から章構成・台本・スライドの自動生成までを一気通貫で求めている点で共通している。

本ツールは、この共通ニーズを1つのSaaSとして実装し、AI研修講師・企業内教育担当者が**テーマと所要時間を入力するだけ**で提供可能な教材ドラフトを得られる状態を目的とする。

## 02. 対象ユーザー・利用形態

マルチユーザーSaaS型として構築する。Supabase Authによるアカウント登録・ログイン機能を実装し、講師ごとにデータを分離する（行レベルセキュリティで保証）。決済・課金機能は本フェーズのスコープ外とし、無料で使える教材作成ツールとしてまず提供する。

- Supabase Auth
- マルチテナント
- RLSでユーザー分離
- 課金機能なし（v1）

## 03. 機能要件

### 3.1 章構成生成

テーマ・所要時間（分）・対象レベル（初級／中級／上級）を入力すると、Claude APIが章構成案を生成する。各章にはタイトル・概要・想定時間を付与し、ユーザーは章の追加・削除・並べ替え・再生成を行える。

### 3.2 章間整合性チェック

全章の構成案を一括でClaude APIに再投入し、重複・矛盾・粒度のばらつきをチェックさせる。指摘があれば章ごとに警告バッジを表示し、ワンクリックで該当章を再生成できるようにする。

### 3.3 台本（読み上げ用）生成

章ごとに読み上げ用の台本をClaude APIで生成する。文体は「です・ます調」に固定し、1章あたり800字以内を厳守する。生成後に文字数をサーバー側でカウントし、超過時は自動的に要約・再生成を行う。章単位での個別再生成・手動編集にも対応する。

### 3.4 スライド画像生成

章タイトル・台本の要約・ユーザーが選択したトーン（ビジネス／カジュアル／ミニマル 等）からプロンプトを組み立て、Gemini API（画像生成対応モデル）で章ごとに1枚のスライド画像を生成する。生成は非同期ジョブとして扱い、章単位の進捗表示・失敗時の自動リトライ（最大2回）を行う。生成画像はSupabase Storageに保存する。

### 3.5 教材管理

教材（プロジェクト）単位でダッシュボードに一覧表示し、詳細画面で章構成・台本・スライドをまとめてプレビューできる。台本テキストと画像一式をZIPでダウンロードする機能を含む。

| 機能 | 使用AI | 入力 | 出力 |
|---|---|---|---|
| 章構成生成 | Claude API | テーマ／所要時間／レベル | 章タイトル・概要・想定時間の配列 |
| 整合性チェック | Claude API | 全章構成案 | 重複・矛盾の指摘リスト |
| 台本生成 | Claude API | 章タイトル・概要・トーン | です・ます調 800字以内の台本 |
| スライド生成 | Gemini API | 章タイトル・台本要約・トーン | スライド画像（1枚） |

## 04. 非機能要件・達成基準

- 60分相当の教材を入力した場合、章構成が **5〜8個** の範囲に収まる（生成プロンプト側でガードレールを設定し、範囲外なら自動調整）
- 台本は **です・ます調** で統一され、**1章あたり800字以内** を機械的に検証する
- スライド画像は **1枚あたり30秒以内** に生成される（Gemini呼び出し＋保存までを計測し、超過時はタイムアウト＋リトライ）
- APIキーはすべてサーバーサイド（Route Handler／Server Action）でのみ使用し、クライアントに露出しない
- 各テーブルにRLSを設定し、ユーザーは自分が作成した教材以外にアクセスできない
- Supabase無料プランが7日間の無操作で一時停止しないよう、定期pingで稼働を維持する

## 05. システム構成

Next.js（App Router）をVercel上でホストし、Claude API・Gemini APIの呼び出しはすべてServer Action／Route Handler経由のサーバーサイド処理とする。認証・DB・ストレージはSupabase、定期実行はVercel Cronで賄う。

```
ブラウザ（講師） ── Next.js App Router / Tailwind + shadcn/ui
        │ HTTPS
        ▼
Vercel（Next.js）── Server Actions / Route Handlers（APIキーはサーバー変数のみ）
        │
        ├─ Vercel Cron ── Supabase定期ping（一時停止防止）／失敗ジョブ再試行
        │
        ├─ Claude API ── 章構成・整合性チェック・台本生成
        ├─ Gemini API ── 章ごとのスライド画像生成
        └─ Supabase ── Auth / Postgres / Storage
```

**技術スタック（確定）**
Next.js 15（App Router）+ TypeScript／Tailwind CSS + shadcn/ui／Anthropic SDK（Claude API）／Google Gen AI SDK（Gemini API）／Supabase（Auth, Postgres, Storage）／Vercel（Hosting, Cron）／GitHub（push時にVercelへ自動デプロイ）

## 06. データベース設計

全テーブルに `user_id`（または親テーブル経由）を持たせ、`auth.uid()` と一致する行のみ操作可能なRLSポリシーを設定する。

| テーブル | 主な列 | 役割 |
|---|---|---|
| `profiles` | id, email, created_at | Supabase Authユーザーの拡張情報 |
| `materials` | id, user_id, theme, duration_minutes, level, tone, status | 教材（プロジェクト）単位のメタ情報 |
| `chapters` | id, material_id, order_index, title, summary, script, char_count, status | 章構成・台本本体 |
| `slides` | id, chapter_id, image_url, prompt, status, generated_at | 章ごとのスライド画像 |
| `generation_logs` | id, material_id, type, status, error_message, created_at | AI呼び出しの成否ログ（コスト・障害の追跡用） |

## 08. AI活用方針

**AIに任せる**
章構成のプロンプト設計と生成、章間の整合性チェック、台本の文体統一・文字数調整、スライド画像プロンプトの自動構築

**人間が判断する**
教材の対象レベル（初級／中級／上級）の最終選択、章の粒度（5〜8章に収めるガードレールの妥当性）、スライド画像のトーン・プリセット設計

コストと品質のバランスを取るため、章構成・整合性チェックのような「精度が重要な工程」はClaude Sonnet 5、台本生成のような「分量が多く定型的な工程」はClaude Haiku 4.5を使い分けるハイブリッド構成を想定する（詳細は09章）。

## 09. コスト試算

60分教材（章7個想定）を1件生成した場合の概算。実際のトークン数はプロンプト設計により変動するため、実装後に `count_tokens` で再計測して補正する。

| 工程 | モデル | 概算コスト／件 |
|---|---|---:|
| 章構成生成＋整合性チェック | Claude Sonnet 5 | 約 $0.02〜0.04 |
| 台本生成（7章） | Claude Haiku 4.5 | 約 $0.02〜0.03 |
| スライド画像生成（7枚） | Gemini 画像生成モデル | 要検証（無料枠の範囲を確認） |
| **合計（画像除く）** | — | **約 $0.04〜0.07 / 教材** |

### 無料枠の考え方

| サービス | 無料枠 | 注意点 |
|---|---|---|
| Vercel（Hobby） | 個人開発なら無料 | 商用提供に切り替える場合はProプラン（$20/月）を検討 |
| Supabase（Free） | DB 500MB／Storage 1GB／Auth 5万MAU | 7日間無操作でプロジェクトが一時停止 → Vercel Cronで週次pingし回避（10章参照） |
| Claude API | 無料枠なし（従量課金） | Haiku 4.5中心の設計でコストを最小化 |
| Gemini API | 無料枠あり（レート制限あり） | 画像生成モデルの無料枠条件は実装時に最新情報を確認 |

## 10. デプロイ・運用自動化

- GitHubリポジトリを新規作成し、mainブランチへのpushでVercelへ自動デプロイする
- 環境変数（`ANTHROPIC_API_KEY`／`GEMINI_API_KEY`／`SUPABASE_URL`／`SUPABASE_ANON_KEY`／`SUPABASE_SERVICE_ROLE_KEY`）はVercel側で管理し、リポジトリには含めない
- Vercel Cronで週1回、Supabaseへの軽量クエリ（例：`SELECT 1`）を実行し、無操作による自動一時停止を防止する

## 11. スコープ外（v1では対応しない）

- Stripe等によるサブスクリプション課金機能
- PDFエクスポートの高度なデザインカスタマイズ（v1はテキスト＋画像のシンプル出力）
- 日本語以外の多言語対応
- チーム・組織単位でのデータ共有機能

## 12. マイルストーン

1. **要件定義書の確認・承認** — 本ドキュメント。修正依頼があれば反映した上で次工程へ。
2. **プロジェクト初期構築** — Next.js scaffold、GitHubリポジトリ作成、Vercel連携、Supabaseプロジェクト作成・スキーマ適用
3. **認証機能** — Supabase Authによるサインアップ／ログイン、RLS設定
4. **章構成生成機能** — テーマ入力→Claude APIで章構成生成、編集UI、整合性チェック
5. **台本生成機能** — 章ごとの台本生成・文字数バリデーション・再生成
6. **スライド画像生成機能** — Gemini APIによる非同期画像生成、進捗表示、Storage保存
7. **教材管理・UI仕上げ** — ダッシュボード、詳細プレビュー、ZIPエクスポート
8. **デプロイ・動作確認** — 本番環境での一連の生成フロー確認、Cron稼働確認

## 13. 要確認事項

実装フェーズに入る前に、以下をご確認・ご用意ください。

**Q1. GitHubリポジトリ名の希望はありますか？**
未指定の場合、既定案として `ai-training-builder` で新規作成します。

**Q2. Supabaseプロジェクトは新規作成でよいですか？**
既存プロジェクト（nishida-portfolio 等）とは別に、本アプリ専用のプロジェクトを新規作成することを推奨します。

**Q3. Claude API／Gemini APIキーの取得**
Anthropic Console（console.anthropic.com）とGoogle AI Studio（aistudio.google.com）でそれぞれ取得し、実装フェーズでVercelの環境変数に設定します。取得手順のご案内が必要であればお申し付けください。

> **承認後の進め方**: 本書の内容に問題がなければ、そのまま実装フェーズ（マイルストーン2）に着手します。修正したい章がある場合は章番号とあわせてご指摘ください。
